# 개선사항 권고문

> 4 데이터셋(D1~D4) 평가 결과를 토대로 한 **3 계층 개선 권고**:
> ① 요구사항 작성 가이드 ② 도구(KoFinRe) 자체 개선 ③ 프로세스·정책 권고

데이터셋:
- **D1** 공공금융 RFP 56건 (비정제) — 219건 요구사항 후보
- **D2** 정형 RFP 30 (REQ_abstract) — 140 sub-req
- **D3** 영문 PURE 79건 (Paska 영문 기준선)
- **D4** 실기업 4모듈 257건 (FA·BG·CI·CM, **익명화**)

---

## 1. 요구사항 작성자용 가이드 (Smell 유형별)

### S1 Non-atomic (복합의무)
**한 요구사항에 둘 이상의 기능 응답 결합**

- ❌ "고객 정보를 조회하여 검증한 후 외부기관에 전송해야 한다"
- ✅ 분리: ① 고객 정보 조회 ② 조회 결과 검증 ③ 외부기관 전송 (각각 별도 요구사항)
- **D4 검출 0건** — 실기업 정의서는 sub-req 분리가 우수 (참고할 만한 모범)

### S2 Incomplete (불완전 — 모든 데이터셋 최다)
**D1 149건 / D4 235건 (91%)** — 가장 심각한 패턴

원인:
- 짧은 키워드 bullet ("데이터 처리, 결과 저장") 형식
- 의무 표현은 있지만 대상 객체 명시 부족
- 조건절(`~경우`)만 있고 결과 부재

권고:
- 모든 요구사항에 **주체·행위·대상·결과** 4요소 명시
- 양식: "**[누가]**가 **[무엇을]**을 **[어떻게]** 해야 한다"

### S3 Ambiguous Term (모호어)
- 금지어: `적절히`, `필요한`, `신속히`, `효율적`, `실시간`, `다양한`
- 대체 — 측정 가능 표현:
  - "실시간" → "응답시간 200ms 이하 (95th percentile)"
  - "신속히" → "5분 이내"
  - "충분한" → "초당 1,000 TPS 이상"

### S4 Weak Obligation (약한의무) — D4 회계 모듈 44건
**평서형 `~한다 / ~된다` 종결**
- ❌ "회계 전표를 자동 생성한다"
- ✅ "회계 전표를 자동 생성**해야 한다**" 또는 "생성**할 수 있어야 한다**"

### S5 Missing Actor (주체 누락) — 모든 데이터셋 상위
**D4 209건 (81%)** — 한국 SI 정의서 관행적 주체 생략

권고:
- 주어 명시: "**시스템은**", "**관리자는**", "**[모듈명]은**"
- 외부 인터페이스: "**[외부기관]으로부터** 정보를 수신해야 한다"

### S6 Missing Quantification (정량 부재)
성능·용량 요구사항에 숫자+단위 필수:
- ❌ "빠른 응답속도를 보장한다"
- ✅ "응답시간 200ms 이하 보장 (95th percentile, 동시접속 1,000명 기준)"

### S7 Undefined Acronym (미정의 약어) — D4 CI(신용정보) 19건
**도메인 약어 첫 등장 시 정의 필수**
- ❌ "AML/CFT 처리"
- ✅ "AML(Anti-Money Laundering, 자금세탁방지)/CFT(Counter-Financing of Terrorism, 테러자금조달방지) 처리"
- **모범: 정의 1회 후 본문 자유 사용**

### S8 Coordination Ambiguity (범위 모호)
"및/또는, 등, 포함, 관련" 사용 시 범위 명시:
- ❌ "신원 인증 등을 지원해야 한다"
- ✅ "신원 인증(아이디/비밀번호, OTP, 인증서) 3종을 지원해야 한다"

### S9 Passive (수동표현)
- ❌ "보고서가 생성되어야 한다"
- ✅ "**[시스템은]** 보고서를 생성해야 한다"

### S10 Unverifiable (검증불가)
- ❌ "안정적으로 운영되어야 한다"
- ✅ "월 가용성 99.9% 이상 보장 (계획정지 제외)"

## 2. 모듈/카테고리별 권고 (D4 실기업 사례 기반)

> 모든 수치는 익명화된 평가 결과

### FA 회계 (95건, smell 96.8% 최고)
| 발견 | 권고 |
|---|---|
| **S4 약한의무 44건** (전체 모듈 중 1위) | 회계 전표·결산 처리 요구사항을 평서형(`처리한다`)에서 의무형(`처리해야 한다`)로 일괄 정비 |
| S2 불완전 90건 | 회계 처리의 주체(시스템·사용자·감사)·대상(분개·계정)·결과(저장·통지) 명시 |
| 추적가능성 강화 | 모든 회계 트랜잭션은 GAAP·K-IFRS 근거 명시 권장 |

### BG 예산 (52건, smell 96.2%)
| 발견 | 권고 |
|---|---|
| S2 불완전 50건 (96%) | 예산 편성·집행·통제 요구사항을 5W1H로 재구조화 |
| S6 정량부재 5건 | 예산 처리량(건수·금액·주기) 명시 |

### CI 신용정보 (57건, smell 93.0%)
| 발견 | 권고 |
|---|---|
| **S7 미정의 약어 19건** (압도적) | 신용정보 도메인 약어 사전(`용어집` 시트) 추가 — 첫 등장 시 풀어쓰기 의무화 |
| S2 불완전 50건 | 신용정보 조회/저장/전송 각 단계의 입출력·예외 명시 |

### CM 공통 (53건, smell 84.9% — 가장 깨끗)
| 발견 | 권고 |
|---|---|
| 모범 사례 — 다른 모듈 참고 | 공통 모듈 작성 패턴을 표준 템플릿화 |
| 잔여 S2 45건 | 그래도 절반 가까이 불완전 — 공통 모듈도 보강 여지 |

## 4. 프로세스·정책 권고

### 4.1 요구사항 검토 프로세스
1. **작성 단계** — 작성자가 KoFinRe `examples/basic_usage.py` 즉시 사용
2. **PR/리뷰 단계** — Smell 임계값 정책 (예: S2/S5 검출률 30% 초과 시 반려)
3. **최종 승인 단계** — Manual Validation 2명 평가자, Cohen's kappa ≥ 0.6 요구

### 4.2 데이터 거버넌스
- 실기업 정의서 활용 시 **항상 익명화 후 외부 공유**
- 보안 등급별 처리:
  - 🟢 공공 게시물 (RFP) — 메타데이터 공유 OK, 원문 재배포 보류
  - 🟡 정형 데이터셋 (REQ_abstract) — 제공자 동의 하에 공개
  - 🔴 실기업 산출물 (D4) — **로컬 only**, 통계만 외부 공유

### 4.3 도구 운영
- 정기 `EXTRACTION_RULES.md` 리뷰 (분기 1회)
- 신규 도메인 적용 시 약어 화이트리스트 추가 → UPDATE.MD 기록
- gold label 데이터셋 누적 (목표: 500건 → 1,000건)

## 6. 작성자 가이드 — 요약 카드

> 작성자에게 1페이지로 배포하기 좋은 핵심 요약.

```
✅ 좋은 요구사항 = 5W1H 완성
   - 누가(주체) / 무엇을(대상) / 언제(조건) / 어디서(범위) / 왜(근거) / 어떻게(방법) / 결과(검증)

❌ 즉시 검수 대상 표현
   - 약한의무: ~한다, ~된다 → ~해야 한다
   - 모호어: 적절히, 신속히, 효율적, 실시간 → 정량값 명시
   - 수동: ~되어야 한다 → 주체 명시 후 능동형
   - 범위: 등을, 및/또는 → 구체 열거

✅ 한 요구사항 = 한 기능
   - "조회 + 검증 + 전송" → 세 개로 분리

✅ 약어 첫 등장 시 풀어쓰기 1회
   - "AML(Anti-Money Laundering)" → 이후 본문에서 자유롭게 "AML" 사용

✅ 성능 = 숫자 + 단위 + 측정 조건
   - "응답시간 200ms 이하 (95th, 동시접속 1,000)"
```
