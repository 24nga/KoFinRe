# CMMI · NCS 기반 요구사항 작성 원칙 vs KoFinRe S1~S19 비교 분석

> 사용자 제공 CMMI(REQM/RD) 9 작성 원칙 + NCS 중분류 5 제약 카테고리 기준으로 KoFinRe smell taxonomy를 재검증하고 갭을 보완한다.

---

## 1. 기준 정의 (사용자 제공)

### 1.1 CMMI 작성 원칙 9종

| 원칙 | CMMI 영역 | 요지 |
|---|---|---|
| 필요성 (Necessary) | REQM | 목표 달성에 필수 요구만 |
| 단일성·원자성 (Singular) | RD | 1 요구 = 1 기능 |
| 명확성·비모호성 (Unambiguous) | RD | 단일 해석·정량 표현 |
| 검증 가능성 (Verifiable) | RD | 시험·분석 가능 |
| 일관성 (Consistent) | REQM | 타 요구·문서와 충돌 없음 |
| 완전성 (Complete) | RD | 기능·예외·경계·규제 누락 X |
| 구현 독립성 (Implementation-free) | RD | "무엇을/왜"에 집중 |
| 추적 가능성 (Traceable) | REQM | 고유 ID + 출처 |
| 현실성·실현 가능성 (Feasible) | RD | 기술·예산·일정 내 |

### 1.2 NCS 중분류 5 제약 카테고리

| 코드 | 카테고리 | 정의 |
|---|---|---|
| TECH | 기술 제약 | 플랫폼·프레임워크·장비 제한 |
| BIZ | 사업·조직 제약 | 예산·일정·정책·인력 |
| COMP | 법·규제 제약 | 개인정보·보안 규정·산업 표준 |
| OPS | 운영·환경 제약 | 운영 시간·데이터센터·기존 시스템 연동 |
| SEC | 안전·보안 제약 | 안전 무결성 등급·접근통제·암호화·감사로그 |

## 2. KoFinRe S1~S15 ↔ CMMI 9 원칙 매핑

| CMMI 원칙 | CMMI 영역 | KoFinRe 매핑 | 충족도 |
|---|---|---|---|
| 필요성 (Necessary) | REQM | — | ❌ **부재** |
| 단일성·원자성 (Singular) | RD | **S1** 복합의무 | ✅ 완전 |
| 명확성·비모호성 (Unambiguous) | RD | **S3** 모호어 + **S8** 범위모호 + **S15** 지시어모호 | ✅ 강함 |
| 검증 가능성 (Verifiable) | RD | **S4** 약한의무 + **S6** 정량부재 + **S10** 검증불가 | ✅ 강함 |
| 일관성 (Consistent) | REQM | — | ❌ **부재** (문서 단위 필요) |
| 완전성 (Complete) | RD | **S2** 불완전 + **S5** 주체누락 + **S14** 수혜자누락 | ✅ 부분 |
| 구현 독립성 (Implementation-free) | RD | **S11** 구현편향 | ✅ 완전 |
| 추적 가능성 (Traceable) | REQM | **S7** 미정의약어 | ⚠️ 부분 (고유 ID 검사 부재) |
| 현실성·실현 가능성 (Feasible) | RD | — | ❌ **부재** |

### 매핑 통계
- ✅ **충족** (5종): Singular, Unambiguous, Verifiable, Complete, Implementation-free
- ⚠️ **부분** (1종): Traceable (약어만 검사, ID 미검사)
- ❌ **부재** (3종): **Necessary, Consistent, Feasible**

→ 사용자 표 9 원칙 중 **5/9 = 56% 충족** (S15까지)

### KoFinRe S12~S13 추가 매핑
- **S12 부정문** → CMMI 표에는 명시 없음 (INCOSE 기준). 일관성·검증가능성에 간접 기여
- **S13 추측표현** → "필요성"과 "검증가능성"의 교집합 (가능하면/필요시 = 검증 불가)

## 3. KoFinRe S1~S15 ↔ NCS 5 제약 카테고리

| NCS 카테고리 | KoFinRe 매핑 | 충족도 |
|---|---|---|
| TECH (기술 제약) | **S11** 구현편향 (반대 방향 — 강제 시 검출) | ⚠️ 반대 |
| BIZ (사업·조직 제약) | — | ❌ **부재** |
| COMP (법·규제 제약) | — | ❌ **부재** |
| OPS (운영·환경 제약) | — | ❌ **부재** |
| SEC (안전·보안 제약) | — | ❌ **부재** |

→ **5/5 NCS 카테고리 모두 KoFinRe 검출 영역에 없음**

NCS 카테고리는 두 가지 관점으로 활용 가능:
1. **제약사항 명시 부재 검출** — RFP에 제약이 명시되어야 하는 영역인데 빠진 경우
2. **제약사항 분류 자동화** — 명시된 제약을 5 카테고리로 자동 라벨링

## 4. 갭 분석 — 부재한 영역

### Gap 1: 필요성 (Necessary) — CMMI REQM
**문제**:
- 사용자 표 예시: "개발팀이 선호하는 특정 폰트를 강제로 사용해야 한다" (불필요한 요구)
- 비즈니스 목적·근거 없는 임의 지시

**검출 가능 신호** (P0):
| 신호 | 예시 |
|---|---|
| 주관·선호 표현 | "선호하는", "임의로", "원하는", "마음에 드는" |
| 강요·강제 표현 | "강제로", "반드시 ~를 사용", "특정 ~로 한정" |
| 비즈니스 근거 부재 | 출처·근거 명시 없는 의무 표현 |

### Gap 2: 일관성 (Consistent) — CMMI REQM
**문제**:
- 동일 기능 다른 정량값 ("응답시간 200ms" vs "500ms")
- 용어·단위 불일치
- 모순 (필수 vs 선택)

**검출 가능 신호** (P3 — 문서 단위, v3.0):
| 신호 | 검출 방식 |
|---|---|
| 동일 용어 다른 정의 | 약어·키워드 클러스터링 |
| 모순 키워드 페어 | "필수" vs "선택", "지원" vs "지원하지 않음" |
| 정량값 불일치 | 같은 메트릭에 다른 수치 |

### Gap 3: 실현 가능성 (Feasible) — CMMI RD
**문제**:
- 사용자 표 예시: "JSON 및 XML 데이터 형식을 모두 지원해야 한다" (실현 가능)
- 반례: "0초 응답시간", "100% 정확도", "완벽한 보안"

**검출 가능 신호** (P1):
| 신호 | 예시 |
|---|---|
| 절대·극단 표현 | "100%", "완벽한", "절대", "0초", "무한대" |
| 비현실적 가용성 | "100% 가용성", "99.9999% 이상" (4 9s 초과) |
| 검증 불가 정확도 | "100% 정확", "오류 0" |

### Gap 4: 추적성 — 고유 ID 부재
**문제**:
- 사용자 표 예시: "COMP-CR-015는 금융지주회사의 내부 통제 기준 문서를 출처로 한다"
- 본문 텍스트에 요구사항 ID(`REQ-001`, `BIZ-CR-010`) 부재

**검출 가능 신호** (P2):
| 신호 | 검출 방식 |
|---|---|
| ID 패턴 부재 | `[A-Z]{2,5}[-_][A-Z]{2,3}[-_]\d{3,5}` 미매칭 |
| 출처·근거 표현 부재 | "근거:", "출처:", "참조:" 부재 |

### Gap 5: NCS 제약 카테고리 명시 부재
**문제**:
- 제약사항 섹션·태그 없이 본문에 흩어져 있음
- 제약의 유형(TECH/BIZ/COMP/OPS/SEC) 분류 부재

**검출·분류 가능** (P2):
- 각 카테고리 키워드 사전으로 본문 분류
- 핵심 카테고리(SEC, COMP) 명시 여부 검사

## 5. 도출된 신규 smell 후보 (S16~S19)

### S16 Necessity-unclear (필요성 불명확) — P0

**정의**: 비즈니스 목적·근거 없는 임의 지시, 주관적 선호 강요
**CMMI**: REQM Necessary
**예시**:
- "개발팀이 **선호하는** 특정 폰트를 **강제로** 사용해야 한다"
- "관리자가 **임의로** 정한 화면 색상을 적용해야 한다"

**검출 패턴**:
```python
NECESSITY_RED_FLAGS = re.compile(
    r'(?:'
    r'선호하는|선호도|마음에\s*드는|원하는|즐겨\s*쓰는'
    r'|강제로|강제\s*적용|임의로|임의의|독단적'
    r'|특정\s*\w+\s*만|단일\s*\w+\s*만\s*사용'
    r'|개발팀이|운영팀이|특정\s*담당자가'
    r')'
)
```

### S17 Feasibility-concern (실현가능성 위험) — P1

**정의**: 기술적·물리적으로 달성 불가능한 절대값·극단 표현
**CMMI**: RD Feasible
**예시**:
- "응답시간 **0초** 보장"
- "**100% 가용성** 유지"
- "**완벽한** 보안 적용"

**검출 패턴**:
```python
INFEASIBLE_EXPRESSIONS = re.compile(
    r'(?:'
    r'100\s*%\s*(?:가용|정확|성공|완벽|보장|만족)'
    r'|0\s*초\s*(?:이내|응답|처리)'
    r'|무한대|무한정|영구\s*보장'
    r'|완벽한|절대적|100\s*프로'
    r'|99\.99{3,}\s*%'  # 5 9s 이상
    r'|장애\s*0|오류\s*0|결함\s*0'
    r')'
)
```

### S18 Missing-traceability-id (고유 ID 부재) — P2

**정의**: 요구사항이 고유 식별자·출처를 명시하지 않아 추적 불가
**CMMI**: REQM Traceable
**예시**:
- 본문: "본 시스템은 사용자 인증을 지원해야 한다." (REQ-ID 없음, 출처 없음)
- 권장: `[FUNC-AUTH-001]` + 출처: "보안정책 §3.2"

**검출**:
- 본문 또는 메타 컬럼에 REQ-ID 패턴 (`[A-Z]{2,5}[-_][A-Z]{2,3}[-_]\d{3,5}`) 부재
- "출처:", "근거:", "참조:" 키워드 부재

### S19 Constraint-category-unclear (제약 카테고리 불명확) — P2

**정의**: 제약사항이 명시되었으나 NCS 5 카테고리(TECH/BIZ/COMP/OPS/SEC) 중 어느 것에 해당하는지 분류 불가
**NCS**: 5 카테고리 분류 원칙
**활용**:
1. 제약사항 자동 라벨링 (보조 모듈)
2. 핵심 카테고리(SEC·COMP) 명시 여부 검사

**카테고리 키워드 사전**:
```python
CONSTRAINT_TECH = {'OS','플랫폼','프레임워크','서버','장비','버전','RHEL','Linux','Windows'}
CONSTRAINT_BIZ  = {'예산','일정','마일스톤','오픈일','마감','한도','상한','초과할 수 없'}
CONSTRAINT_COMP = {'개인정보','GDPR','보안 규정','법령','준수','PII','금융감독','전자금융감독'}
CONSTRAINT_OPS  = {'24시간','365일','무중단','정기점검','운영시간','데이터센터','연동'}
CONSTRAINT_SEC  = {'MFA','다중인증','암호화','AES','RSA','감사로그','접근통제','권한'}
```

## 7. CMMI 체크리스트 자동 매핑 — 작성자 가이드

KoFinRe smell 검출 결과를 사용자 표의 CMMI 체크리스트 질문에 자동 매핑해 작성자에게 제시 가능:

| 검출 Smell | CMMI 체크리스트 질문 | 권장 작성 변경 |

| S2/S5/S14 | 모든 측면(기능·예외·성능·보안)이 누락 없이 정의되었는가? | 5W1H 보강 |
| S3/S8/S15 | 모든 용어가 단일 해석되는가? | 정량값·정의 명시 |
| S4/S6/S10/S17 | 충족 여부를 측정 가능한 기준으로 확인 가능한가? | 정량 기준 |
| S11 | 특정 기술 선택을 강제하고 있지 않은가? | 기술 독립 표현 |
| S12 | 능동·긍정 표현으로 작성되었는가? | 차단·허용 능동형 |
| S13 | 추측·조건부 표현 제거되었는가? | 명확 조건 |
| **S16** | **이 요구가 비즈니스 목표 달성에 필수적인가?** | **근거·필요성 명시** |
| **S18** | **고유 식별자 + 상위 요구/출처와 연결되었는가?** | **REQ-ID + 출처 명시** |
| **S19** | **제약사항의 카테고리(TECH/BIZ/COMP/OPS/SEC)가 명확한가?** | **카테고리 태그** |

## 10. 결론

### CMMI 측면
- KoFinRe S1~S15가 RD(Requirements Development)는 강하게 커버
- **REQM(Requirements Management) 영역의 Necessary·Consistent·Traceable 일부 부재**
- v2.8 S16/S18 추가로 REQM 강화

### NCS 측면
- 제약사항 5 카테고리 검출·분류는 완전 부재
